9. ~~**Budget de reasoning par tour**~~ — **LIVRÉ 2026-07-11** (`4cb1d15`, src/reasoning-budget.ts). Vérifié live (budget 400 chars, scratch :5399) : checkpoints 1/2 et 2/2 tracés 🧠 dans le transcript, le 27B répond au menu par « (b) answer now. » + verdict complet. Sièges locaux uniquement, `PIPELINE_REASONING_BUDGET` (défaut 25K chars, 0 désactive), continues bornés 2/tour, abort+re-prompt même session (le thinking partiel reste en contexte : continue reprend, ne repart pas). Design d'origine — le Qwopus stock n'a pas de thinking level et tombe en attracteur d'overthink (« actually, let me reconsider » → « wait, I've already verified this ») sur les consignes ouvertes. Datum discussion 57 : 130K chars de reasoning pour 5,7K de réponse (tour 5), tour quasi muet à 44K/33 chars (tour 27), contre 3,3K chars quand la consigne du planner est une checklist mécanique (tour 35). Un system prompt seul n'en sort pas : l'instruction est à 100K tokens de la boucle et n'influence que l'*entrée* en réflexion, pas la sortie. Fix harness (design affiné 2026-07-11, dax) : **checkpoint, pas guillotine** — encadrer le raisonnement, pas le contraindre (un deep reasoner cloud type Sonnet 5 dépense légitimement beaucoup ; lui dire « stop » est faux). La room streame déjà le reasoning (`liveReasoning`) ; au budget dépassé, abort + injection : « you have spent your allotted reasoning budget (N tokens) — choose: (a) continue (state in ONE line what remains to resolve — grants another N), (b) answer now with what you have, (c) ask ONE question to @planner/@auditor/user ». Garde-fous : **continues bornés** (2/tour, sinon la boucle survit au checkpoint en répondant « continue » indéfiniment) ; justification de continue qui se répète = signature de boucle → bascule (b)/(c) ; **budget par siège via le roster** (local serré ~25K chars, cloud reasoner large/illimité — le tag local/cloud porte le bit) ; **chaque checkpoint laisse une trace transcript** (🧠 checkpoint 1/2 + la ligne « what remains ») — zéro burn muet, même invariant que zéro hop muet ; formulation positive (éléphant rose : jamais « stop overthinking »). Ce qui casse la boucle n'est pas la coupe, c'est l'entrée d'information NOUVELLE dans le contexte. Cohérent post-circuit-breaker : bornage de ressource, pas détection de motif. Même famille que docs/resource-aware-supervision.md. Levier immédiat en attendant : le style de dispatch du planner (checklist mécanique > consigne ouverte).
10. **Fused seats — chapeaux de rôle sur contextes partagés** (dax + Fable, 2026-07-11, design : docs/fused-seats.md). Persona = chapeau (prompt + allowlist par tour), contexte = siège partagé par cluster : maker (builder+tester), orchestrateur (planner+scribe), **auditor stateless — ne fusionne jamais** (audit sur pièces : le contexte partagé est une contamination, pas une économie). Changement à la couche session uniquement (`persona → seat` sur la clé) : routing, traces et supervision inchangés. Gains attendus : cache de préfixe llama-server (2 préfixes au lieu de 7 sur `--parallel 1`), mort de la taxe de re-dérivation (le handoff intra-siège devient un changement de chapeau à zéro token — datum tester : 83 calls/130K chars de reasoning à re-établir l'état du builder). Phase 2 éventuelle : LoRA par siège (jamais de merge complet — 2×16GB ne tiennent pas dans 24GB), gatée sur drift de chapeau mesuré. Distinct d'OpenRouter Fusion (fan-out + synthèse de *sorties*, stateless, ensemble opaque) : ici on fusionne les *contextes de travail*, observable et moins cher, pas les réponses.

## Décisions actées (ne pas rouvrir sans nouveau contexte)

- **Supervised routing phase 1 livré** (2026-07-10, board 7/7, 1140/1140, audit clos). Le planner devient superviseur du dispatch au lieu de hop dans la chaîne : mode `supervised` additif (zéro rename, zéro migration), `supervisorAgent` (défaut planner), décision stateless en session éphémère (modèle du persona superviseur, tool unique `route_decision`, thinking low). Sémantiques actées : **refuse = return-to-sender** avec cap anti-ping-pong (re-proposition identique → fallback/drop, non supervisée — trappe de sortie) ; **dégradation = dispatch** (tout résultat non-décision → accept + notice, jamais stall ni drop) ; **plan-owner routing non supervisé** (le superviseur ne juge pas son propre plan) ; **auto-accept** des sets entièrement proposés par le superviseur, set mixte jugé entier ; **double `route_decision` = terminate** (écart délibéré vs handoff : une décision capturée n'a rien à réessayer). Design doc + retro : docs/supervised-routing.md.

- **Boucle d'orchestration fermée** (commit `cbffaa2`, release 0.1.22, 2026-07-09). spawn_room n'est plus fire-and-forget : rapport automatique dans la room parente à la résolution du goal (tour du spawner déclenché, passif si pause), `ask_orchestrator` dans toute sub-room spawnée (pause type ask_user, marche dans les boucles goal-eval), `answer_room` pour répondre/reprendre. Limite connue : le ParentLink est en mémoire — après restart serveur, une sub-room restaurée ne rapporte plus (check_room reste).

- **Task board partagé livré** (commit `970841e`, release 0.1.21, 2026-07-09). Deux systèmes assumés : plans = contrat d'ingénierie global + routing ; board = couche d'orchestration vivante par room, visible en continu (TUI Ctrl+P + résumé sous roster, panneau sidebar web). Tools gated sur la présence du board, pas sur l'allowlist persona — les personas persistées d'avant la feature les reçoivent sans migration.

- **Héritage des capacités persona dans les presets** (2026-07-10, board 10/10, 1075/1075 + 111/111 TUI). Sémantique : `skills` absent = hérite du SEED de même id, présent (même `[]`) = respecté ; `tools` n'hérite **jamais** (liste explicite = intention explicite — un preset qui restreint un agent le fait exprès). Implémenté en étendant `rehydratePrompts` → `rehydrateSeedFields` (module pur `src/preset-hydration.ts`), couvre `POST /api/rooms` ET `spawn_room` via `provisionRoom`. **Strip-si-identique pour `skills`** : à la sauvegarde, `skills` est retiré uniquement s'il est identique au SEED — non-modifié → absent sur disque → hérite en continu (anti-drift permanent) ; custom ou opt-out `[]` → diffère → préservé. SKILL.md orchestrator assoupli au passage : poll libre, note restart (ParentLink en mémoire), section parallélisme hétérogène (compter les backends, pas les rooms). **TUI** : HeaderDivider entre le bloc header et la conversation.
- **« User dispatch outranks supervisor » + invariant zéro-hop-muet** (2026-07-11, re-run #3 PASS). Un handoff vers un agent déjà pré-queué par mention utilisateur **coalesce sans passer par le superviseur** — cohérent avec les autorités déjà actées (plan-owner routing non supervisé, auto-accept des sets proposés par le superviseur) : le superviseur gate le jugement des agents, pas les directives humaines ; le router quand même serait de la complexité pour affaiblir l'utilisateur. **Mais en supervised, aucun hop ne reste muet** : chaque hop montre soit un glyphe superviseur (✓ accept / ↪ transfer / ✗ refuse / ⚠ degrade), soit une ligne de coalescence `≡` system-authored. Libellé `≡` cause-neutre (« already queued — handoff coalesced (not supervised) ») : la condition `target.includes(p)` ne distingue pas un dispatch utilisateur d'un accept superviseur antérieur dans le même drain (auditor, #11). Coût du silence documenté : 3 runs live et des jours de diagnostic parce qu'une coalescence légitime était indistinguable d'un bypass.
- **Règle module-first** (2026-07-11). Toute nouvelle feature naît dans son propre module `src/<feature>.ts` ; room.ts/server.ts ne reçoivent que le point de câblage. Opposable en audit. Motivé par la croissance de room.ts (1795 → 2458 lignes en une semaine) — chaque feature rewire à travers room.ts plus vite que les extractions ne retirent.
- **Norme d'audit** (2026-07-10). Tout changement sous `src/` reçoit une passe auditor avant clôture de plan. Le tester valide le comportement ; l'auditor valide la conception. Le premier round a sauté l'auditor — le tester a vérifié que ça marchait, mais l'auditor a trouvé un défaut de design (self-heal one-shot pour `skills`). Le second round a fermé les 3 findings + TUI divider.
- **Routing @-mention** : seul `@<name>` déclenche — parler d'un agent par son nom est sûr. Clarifié par dax 2026-07-08.
